# 串口通讯理解笔记

## 1. 串口通讯的本质

串口（UART/RS232/RS485）是**异步串行通讯**，核心就一件事：

**发送方和接收方约定好一套"节奏和规则"，然后一位一位地传数据。**

就像两个人用对讲机通话——双方必须同时按住同一个频道、说同一种语言，否则就乱套。

---

## 2. 串口四件套（必须双方完全一致）

### 波特率（Baud Rate）
- 单位：bps（bits per second）
- 本质：双方约定的"通讯速度"，即每秒传多少个比特
- 类比：对讲机的语速
- 常见值：9600 / 19200 / 38400 / 57600 / 115200 / 230400 / 460800 / 921600
- **关键：9600 表示每秒传 9600 个比特，不是 9600 个字节**

### 数据位（Data Bits）
- 每个"字符"（byte）用多少比特来表示
- 常见值：8（最常用，足够表示一个完整字节）、7、6、5
- 8 位数据位 = 传输 ASCII 或任何 0-255 的数据
- 7 位数据位 = 只够传标准 ASCII（0-127）

### 停止位（Stop Bits）
- 每个字符传完后，加 1~2 个"空闲位"，告诉对方"这个字符结束了"
- 常见值：1（最常用）、1.5、2
- 1.5 位是历史遗留，正常项目用 1 位就够了

### 校验位（Parity）
- 简单的错误检测机制
- 种类：None（无校验）/ Odd（奇校验）/ Even（偶校验）/ Mark（固定为1）/ Space（固定为0）
- 原理：发送时额外发 1 位，让整个字符帧中 1 的个数为奇/偶
- **注意：加了校验位，数据帧就从 8+1=9 位变成了 10 位/字符，实际传输效率降低**

---

## 3. 一个字符的传输时序

以 9600 N 8 1 为例，传一个字符 'A'（ASCII 65 = 0x41）：

```
     起始位   bit0 bit1 bit2 bit3 bit4 bit5 bit6 bit7   停止位
    +-------+----+----+----+----+----+----+----+----+-------+
    |   0   |  1 |  0 |  0 |  0 |  0 |  0 |  1 |  0 |   1   |
    +-------+----+----+----+----+----+----+----+----+-------+
              'A' = 0x41 = 01000001
```

- 空闲时线路为 1（MARK 状态）
- 起始位固定为 0（SPACE），用来同步
- 先传最低位（LSB first）
- 停止位固定为 1

**传一个字符实际需要：1起始位 + 8数据位 + 1停止位 = 10 比特**
**9600 波特率下，每秒最多传 960 个字符（约 9600/10）**

---

## 4. 全双工 vs 半双工

- **全双工（RS232）**：TX 和 RX 两根线，可以同时发、同时收，互不影响
- **半双工（RS485）**：只有一根数据线，同一时刻只能发或只能收，需要控制方向切换

我们软件模拟的是全双工场景（RS232）。

---

## 5. 串口服务器的作用

真实串口服务器是"桥接器"：

```
[网络客户端] <--TCP/IP网络--> [串口服务器] <--RS232--> [真实串口设备]
```

- 左边：网络侧，客户端通过 Telnet（或 TCP 直连）连接
- 右边：串口侧，连接真实的传感器、PLC、单片机等
- 串口服务器负责把网络数据透传到串口，并把串口应答透传回网络

**关键：串口服务器本身不处理业务数据，它只做协议转换和透传。**

---

## 6. Telnet + RFC2217 在这其中的角色

### 为什么需要 Telnet？
串口服务器通过网络提供服务，但网络是 TCP，数据是字节流，没有"帧"的概念。
Telnet 协议（RFC854）定义了一套文本交互的方式，包括：
- IAC (0xFF) 转义序列
- DO/DONT/WILL/WONT 选项协商

但原始 Telnet 只有"远程回显"等功能，没有专门针对**串口参数协商**的定义。

### 为什么需要 RFC2217？
RFC2217 在 Telnet 选项协商框架上，扩展了**COM_PORT_OPTION (0x21)**，允许：
- 客户端动态告诉服务端"我现在把波特率改成 115200"
- 服务端把这个参数应用到实际的串口硬件上

**本质上：RFC2217 就是让客户端能"遥控"串口服务器的串口参数。**

---

## 7. 软件模拟的含义

软件版串口服务器，把"透传到真实串口硬件"这截换成了"规则引擎"：

```
[PuTTY + RFC2217] <--Telnet+RFC2217协议--> [软件串口服务器] <--规则引擎--> [无物理设备]
```

- 网络侧完全一样，PuTTY 感受不到差异
- 串口侧，我们用"仿真规则"模拟真实设备的应答
- SSCOM 这类工具换成任何支持 Telnet/RFC2217 的客户端都可以

---

## 8. 几个容易混淆的点

### 8.1 波特率 115200 是不是固定的？
不是。RFC2217 的 SETBAUD 可以动态改变波特率。PuTTY 通过 RFC2217 改了波特率后，软件服务端应该记录这个新值（虽然模拟环境下没有真实硬件用到它）。

### 8.2 Telnet 和 TCP 直连有什么区别？
- **TCP 直连**：纯字节流，没有任何协议包装
- **Telnet**：在 TCP 基础上加了一层 Telnet 协议（IAC 转义、选项协商）
- RFC2217 是 Telnet 的扩展选项，必须先建立 Telnet 连接才能用

PuTTY 连接串口服务器时通常选 "Telnet" 模式，而不是 "Raw TCP"。

### 8.3 为什么需要 IAC 转义？
0xFF 是 Telnet 命令字节，正常数据如果出现 0xFF 就可能被误解释为命令。
所以数据中的 0xFF 必须用 **IAC IAC**（两个 0xFF）来表示，接收方遇到连续两个 0xFF 会还原成一个 0xFF 数据。

### 8.4 回显模式 vs 指令模式
- **回显模式**：收到什么原样返回（类似"回环测试"）
- **指令模式**：根据预设规则匹配后返回特定响应
- 两者可以并存：先匹配指令规则，无匹配则回显

## 9. 实际调试串口服务器时的常见问题

| 问题 | 原因 |
|------|------|
| 连接成功但发数据没响应 | 规则没有匹配，或规则 reply 为空 |
| RFC2217 协商不成功 | 客户端没发送 IAC DO COM_PORT_OPTION，服务端没回复 WILL |
| 波特率改了但没生效 | 模拟环境不实际用波特率，只记录日志 |
| PuTTY 连接被拒绝 | 端口号填错，或服务端没启动 |
| 收到乱码 | 编码问题（UTF-8 vs GBK），或两端字符集不一致 |
